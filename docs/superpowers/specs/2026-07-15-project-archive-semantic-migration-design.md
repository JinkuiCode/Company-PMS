# 项目档案字段语义化迁移设计

## 背景

项目档案当前使用 `product_line`、`product_type` 等历史字段名，页面分别显示为产品线和产品类型；项目进度总表中又存在客户、设备系列、序列号等重复维护字段。业务已确认系统尚未正式上线，当前数据均为测试数据，可以执行一次完整的语义化调整。

本次调整需要同步修改数据库模型、接口字段、字段目录、字段规则、枚举管理、角色产品范围、项目进度引用字段和前端页面，避免只改页面文字形成新的技术债务。

## 目标

- 项目编号改为可编辑文本，并保持全局唯一。
- 项目名称保持必填，并增加全局唯一约束。
- 项目档案新增客户文本字段，暂不接入 OA 客户档案。
- 产品线正式更名为产品类别，后台字段和页面文字保持一致。
- 产品类型正式更名为设备系列，后台字段和页面文字保持一致。
- 项目档案新增序列号文本字段，并保持全局唯一。
- 产品类别、设备系列等业务枚举使用稳定数字流水，不使用枚举表数据库主键。
- 项目进度中的上述字段改为项目档案只读引用，继续遵守“引用字段不可在消费页面修改”的项目规则。

## 非目标

- 本次不接入 OA 客户档案，客户仍由用户输入文本。
- 本次不建设金蝶内码、历史项目编码或 ERP 编码重命名能力。
- ERP 同步状态继续使用 `success`、`failed`、`pending` 协议值，不参与业务枚举数字化。
- 业务表不直接外键关联 `sys_dict_item.id`。

## 字段模型

### 项目档案

`pms_project_archive` 的业务模型调整为：

| 字段 | 类型 | 规则 |
| --- | --- | --- |
| `project_code` | 文本 | 必填、可编辑、全局唯一 |
| `project_name` | 文本 | 必填、可编辑、全局唯一 |
| `customer` | 文本 | 可编辑、允许为空 |
| `product_category` | 整数 | 产品类别枚举、允许为空 |
| `equipment_series` | 整数 | 设备系列枚举、允许为空 |
| `serial_no` | 文本 | 可编辑、允许为空、非空值全局唯一 |
| `status` | 整数 | 档案状态枚举 |
| `manager_id` | 整数 | 负责人用户 ID |
| `plan_start_date` | 日期时间 | 计划开始 |
| `plan_end_date` | 日期时间 | 计划结束 |

旧字段 `product_line` 和 `product_type` 在模型、Schema、接口和前端中分别由 `product_category` 和 `equipment_series` 替代。已有开发数据库迁移完成后不再读取旧字段；全新数据库只创建新字段。

### 唯一性规范

- 项目编号、项目名称和序列号保存前统一去除首尾空格。
- 项目编号和序列号使用统一大小写标准化键，`SN001` 与 `sn001` 视为重复。
- 项目名称按去除首尾空格后的完整名称校验唯一。
- 项目编号和项目名称使用非空唯一索引；序列号使用仅约束非空值的唯一索引。
- 后端在写入前主动检查并返回包含 `field_key` 的结构化 `409`，数据库唯一索引作为并发写入的最终防线。
- 编辑时排除当前档案自身；任何冲突都不产生部分更新。

### 项目进度冗余字段

`pms_project` 当前仍保存项目编号、项目名称和产品类别的列表查询副本。修改项目档案时，在同一数据库事务中同步更新所有关联项目，避免项目档案与项目进度展示不一致。

项目进度总表注册表中的以下字段改为档案来源、只读：

- `customer`
- `product_category`
- `equipment_series`
- `serial_no`
- `project_code`
- `project_name`

项目总表明细 JSON 不再接受这些字段的新写入。迁移时如果存在旧明细值，优先迁入对应档案；同一档案关联的多个项目存在冲突值时终止迁移并输出冲突清单，不自动选择数据。

## 枚举数字流水

### 枚举定义

- 新枚举编码为 `product_category` 和 `equipment_series`。
- `archive_status`、`project_status`、`task_status` 保持现有数字值。
- `erp_sync_status` 保持系统协议字符串，不进入本次数字化范围。

### 流水规则

- 每个枚举分类独立从 `1` 开始编号。
- 新增可配置枚举时，系统按该分类历史最大数字加一生成存储值。
- 用户只维护显示名称、排序和启停状态，不手工输入存储值。
- 数字存储值生成后不可修改；删除后不复用。
- `sys_dict_item.item_value` 作为通用枚举表兼容字段保存数字字符串，项目档案和项目表中的业务字段使用整数列。
- 流水号不是 `sys_dict_item.id`，因此在不同数据库环境中仍可保持一致。

### 测试数据迁移

旧枚举按当前排序和 ID 稳定映射：

- 产品类别：`Bench=1`、`光伏=2`、`Single=3`、`HOTSPM=4`。
- 设备系列：`链式=1`、`槽式=2`。
- 其他自定义值继续按排序和 ID 依次编号。

迁移需要同时更新项目档案、项目进度冗余字段以及角色产品类别范围。

## 权限与字段规则

- 角色字段 `product_lines` 在模型、Schema、接口和前端语义化更名为 `product_category_ids`，保存逗号分隔的稳定数字编号。
- `/api/auth/me` 和产品范围接口返回 `product_category_ids: number[] | null`。
- 数据范围查询和写入校验使用产品类别数字编号，不再比较显示名称。
- 项目档案字段规则新增客户、产品类别、设备系列和序列号。
- 项目编号保持代码锁定为可见、可编辑、必填，业务管理员不能把核心项目编号配置为隐藏或不可编辑。
- 项目进度中的档案引用字段继续锁定为不可编辑。

## API 与前端交互

- 项目档案创建、更新、列表和下拉接口统一返回新字段名。
- 项目档案创建弹窗和编辑抽屉按新字段模型展示。
- 编辑抽屉只有点击字段值才进入编辑；保存仍使用统一底部操作栏。
- 项目编号、项目名称和序列号在失焦或统一保存时执行唯一性检查，冲突时在对应字段下显示中文错误。
- 列表列、筛选字段和列设置同步新增客户、序列号，并使用“产品类别”“设备系列”中文名称。
- 项目进度的筛选、动态列、详情抽屉和档案选择展示新字段，档案引用字段只读。
- 枚举管理新增值时不再显示可编辑的存储值输入框，改为展示“保存后自动生成数字编号”。
- 浏览器列偏好存储键升级版本，自动清理旧 `product_line`、`product_type` 列配置。

## ERP 行为

PMS 继续只使用当前项目编号和项目名称查询、创建或更新金蝶项目档案。本次不保存金蝶内码，也不兼容正式生产历史编码。由于系统尚未上线，修改项目编号后按新编号执行后续测试同步即可。

## 数据迁移与失败处理

- 初始化升级先读取旧枚举和旧业务值，生成确定的数字映射，再写入新列。
- 唯一索引建立前扫描项目编号、项目名称和序列号冲突。
- 发现重复、无法映射的枚举或项目总表明细冲突时，迁移整体回滚并输出明确记录，不静默丢弃数据。
- SQLite 本地开发和 MSSQL 2017 使用相同业务映射规则，并分别使用兼容的建列与索引语句。
- 所有项目档案写操作继续接入统一操作日志，日志中文字段名使用更新后的字段目录。

## 测试策略

- 后端契约测试覆盖新模型、Schema、字段目录、字段规则和枚举注册表。
- 验证项目编号、项目名称、序列号的新增冲突、编辑冲突、大小写和空值行为。
- 验证数字流水自动生成、不可修改、不复用及并发保护。
- 验证旧产品线、产品类型、角色范围和项目总表明细的迁移映射。
- 验证档案修改能同步更新关联项目，并完整写入一条操作日志。
- 验证项目进度中的客户、产品类别、设备系列和序列号均为只读引用。
- 前端运行项目档案、项目进度、字段规则、枚举管理、数据字典及 UI 契约测试，并执行 `npm run build`。
- 浏览器检查新增弹窗、编辑抽屉、唯一性错误、筛选、列设置和枚举新增流程。

## 实施边界

本次以语义一致性优先，不保留旧 API 字段名作为长期双写接口。现有测试数据库通过一次迁移升级；尚未上线的环境允许在迁移失败时清理测试数据后重新初始化，但程序仍需给出明确失败原因。
